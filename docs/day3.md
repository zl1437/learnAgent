# Day 3：CoT 思维链——从「直觉闪回」到「逻辑推理」

## 目标

掌握 **CoT（Chain of Thought，思维链）** 提示技术，用于复杂逻辑、多步运算或需要隐含推理的任务。让模型不直接「猜」答案，而是先拆解步骤、再输出结果。

---

## 1. 核心理论学习（约 30 分钟）

### 什么是 CoT？

- **现象**：复杂数学题或逻辑陷阱题时，模型可能靠「直觉」预测直接答错。
- **原理**：要求模型 **Step by Step（一步步思考）**，把问题拆成中间推理步骤。
- **本质**：用更多输出 Token 换取更长的「思考链」，通常能提高逻辑准确度。

### CoT 的两种常见写法

| 方式 | 说明 |
|------|------|
| **Zero-shot CoT** | 在指令末尾加一句，如「请一步步思考并说明理由」。 |
| **Few-shot CoT** | 在示例里不仅给「输入 -> 输出」，还展示中间的**推理过程**。 |

---

## 2. 定量化实操任务（约 90 分钟）

### 场景

开发「智能报销助手」：根据用户一段话，**计算总金额**并判断是否超标（示例规则：每人每天餐补不超过 **100 元**，需按**人数**折算人均后再对比）。

### 步骤 A：对比实验（直觉 vs 逻辑）

- **任务**：测试模型在**隐藏规则**（按人头算上限）下的表现。
- **输入（Zero-shot）**：

```text
用户：我们一共 3 个人，中午吃了 150 元，晚上吃了 180 元。请问总共报销多少？有没有超标？只返回 JSON。
```

- **观察**：是否漏算总额？是否忽略「按人头 / 每日上限」等隐含条件？

### 步骤 B：Zero-shot CoT（先拆解再输出）

- **POST URL**：`https://api.deepseek.com/v1/chat/completions`
- **JSON Body** 示例：

```json
{
  "model": "deepseek-chat",
  "messages": [
    {
      "role": "system",
      "content": "你是一个财务审查助手。请先进行逻辑拆解（计算总额、人均金额、对比阈值），最后再输出 JSON。请一步步思考。"
    },
    {
      "role": "user",
      "content": "我们一共 3 个人，中午吃了 150 元，晚上吃了 180 元。上限是每人每天100元。请问报销总额及是否超标？"
    }
  ]
}
```

> 请谨慎保管 API Key，勿提交到公开仓库。

### 步骤 C：Few-shot CoT（展示「思考样板」）

在 Prompt 中同时给出**推理过程 + 最终 JSON**，约束最强，适合 Agent 工程中的稳定格式。

**示例对话结构：**

```text
用户：2个人，午餐80，晚餐140。上限人均100。
输出：
总金额：80 + 140 = 220 元。
人均金额：220 / 2 = 110 元。
判断：110 > 100，超标。
结果：{"total": 220, "over_limit": true}

用户：3个人，午餐150，晚餐120。上限人均100。
输出：
总金额：150 + 120 = 270 元。
人均金额：270 / 3 = 90 元。
判断：90 ≤ 100，未超标。
结果：{"total": 270, "over_limit": false}
```

将上述「样板」放进 `system` 或与 `user` 配套的 Few-shot 段落后，再发真实用户输入做对比测试。

## 3. 今日定量化完成指标

- [ ] **逻辑可追踪**：回复中包含清晰计算步骤，且 **3 组不同数值**下结果 **100%** 正确。
- [ ] **隐喻理解**：当用户说「我们加上王总一共 4 人」时，能准确得到 `count: 4`（或等价字段）。
- [ ] **结构化提取**：推理过程之后，仍能稳定输出 **合法 JSON**（便于 `JSON.parse`）。

## 4. 前端开发思考：如何展示「思考过程」

在前端 Agent 中，往往**不希望把冗长推理直接展示给最终用户**，但又要能调试。

### 技巧 1：分隔符或标签拆分

要求模型用 `---` 或 `<reasoning>...</reasoning>` 包裹推理过程；前端用字符串 `split` 或解析标签，**只把最后一段 JSON 用于 UI**，前面保留给日志或调试面板。
```
{
  "model": "deepseek-chat",
  "messages": [
    {
      "role": "system",
      "content": "你是一个财务审查助手。请先进行逻辑拆解（计算总额、人均金额、对比阈值），最后再输出 JSON。请一步步思考。示例 用户：2个人，午餐80，晚餐140。上限人均100。 <reasoning> 总金额：80 + 140 = 220 元。 人均金额：220 / 2 = 110 元。 判断：110 > 100，超标。</reasoning>{\"total\": 220, \"over_limit\": true} 用户：3个人，午餐150，晚餐120。上限人均100。 <reasoning>总金额：150 + 120 = 270 元。 人均金额：270 / 3 = 90 元。 判断：90 ≤ 100，未超标。</reasoning>{\"total\": 270, \"over_limit\": false}"
    },
    {
      "role": "user",
      "content": "我们加上王总一共 4 人,午餐180，晚餐240。上限人均100"
    }
  ]
}
```

### 技巧 2：DeepSeek-R1 / `deepseek-reasoner`

若使用 **`deepseek-reasoner`** 等带推理能力的模型，响应里可能出现 **`reasoning_content`**（或类似字段，以实际 API 文档为准）。可在 Postman 中切换 `model` 对比返回结构差异，再决定前端如何展示「思考」与「最终答案」。
```
"message": {
                "role": "assistant",
                "content": "<reasoning>总金额：午餐180元 + 晚餐240元 = 420元。  \n人均金额：420元 / 4人 = 105元。  \n判断：105元 > 100元，超标。</reasoning>{\"total\": 420, \"over_limit\": true}",
                "reasoning_content": "首先，用户查询是：“我们加上王总一共 4 人,午餐180，晚餐240。上限人均100。”\n\n我需要按照指示进行逻辑拆解：计算总额、人均金额、对比阈值，然后输出 JSON。\n\n步骤：\n\n1. **计算总额**：午餐费用是180元，晚餐费用是240元。所以总金额 = 180 + 240。\n\n   - 180 + 240 = 420元。\n\n2. **计算人均金额**：总共有4个人。所以人均金额 = 总额 / 人数。\n\n   - 人均金额 = 420 / 4 = 105元。\n\n3. **对比阈值**：上限是人均100元。判断人均金额是否超过阈值。\n\n   - 105元 > 100元，所以超标。\n\n现在，逻辑拆解完成。最后输出 JSON。JSON 应该包含 \"total\" 和 \"over_limit\" 字段。\n\n- \"total\": 420\n\n- \"over_limit\": true（因为超标）\n\n在示例中，JSON 是 {\"total\": 220, \"over_limit\": true} 或类似格式。\n\n所以，输出应该先有 reasoning 部分，然后是 JSON。\n\nreasoning 部分：我需要一步步写出思考过程。\n\n用户消息是中文，所以我可以保持推理部分为中文，但示例中是中文推理，所以我应该用中文写推理。\n\n示例推理：<reasoning>总金额：80 + 140 = 220 元。 人均金额：220 / 2 = 110 元。 判断：110 > 100，超标。</reasoning>\n\n类似地，我应该这样写。\n\n推理：\n\n- 总金额：午餐180元 + 晚餐240元 = 420元。\n\n- 人均金额：420元 / 4人 = 105元。\n\n- 判断：105元 > 100元，超标。\n\n然后 JSON：{\"total\": 420, \"over_limit\": true}\n\n最后输出格式：先 reasoning 标签，然后 JSON。\n\n确保 JSON 是有效的。"
            },
```
